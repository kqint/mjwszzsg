# 修复总结

## 背景

目标：让老游戏 APK（`mjwszzsg.apk`，Cocos2dx 引擎，360 渠道，约 2014 年）在 Android 11 设备 (Realme RMX3031) 上运行。

- 包名：`com.gamedo.v360`
- 仅 `armeabi` (32-bit ARM) 原生库：`libgame.so` (7.7MB, 内嵌 libcurl 7.26.0 + Cocos2dx)、`libidentifyapp.so`、`libcasdkjni.so`
- 集成 7 个 SDK：360 safiap、mm purchasesdk、CMGame billing、咪咕 mmwlan、multimode_billing、cn.game189.sms、com.a (渠道SDK)
- 原始无 targetSdkVersion 声明

## 全部有效修改（14 处）

### Java/Smali（7 处）

#### 1. targetSdkVersion=22（AndroidManifest.xml）

```xml
<uses-sdk android:minSdkVersion="8" android:targetSdkVersion="22"/>
<application android:requestLegacyExternalStorage="true" android:usesCleartextTraffic="true">
```

**效果**：一次性避开 API 23-29 的全部破坏性变更：
- getDeviceId() SecurityException（29+ 处调用点全部安全）
- Apache HTTP 类缺失
- Non-SDK 接口灰名单
- 运行时权限（安装时自动授予）
- 分区存储（requestLegacyExternalStorage 在 Android 11 上对 targetSdkVersion≤29 生效）

#### 2. 禁止 .so 压缩（apktool.yml）

```yaml
doNotCompress:
- so   # 防 UnsatisfiedLinkError
```

#### 3. StrictMode 放行主线程网络（sanguoAndroid360.smali）

在 `onCreate()` 开头注入：
```java
StrictMode.ThreadPolicy policy = new StrictMode.ThreadPolicy.Builder().permitAll().build();
StrictMode.setThreadPolicy(policy);
```
**效果**：NetworkOnMainThreadException → 修复。

#### 4. SmsSdk.init() 中 4 个 telephony 调用 stub

```smali
; getLine1Number / getSubscriberId / getDeviceId / getSimSerialNumber
; 每个调用替换为:
const-string v21, ""
```
**效果**：SELinux 阻断 `iphonesubinfo1` 服务导致的 binder 超时 → 消除。

#### 5. sendGameInfo() stub（SmsSdk.smali）

```smali
.method public sendGameInfo(Ljava/lang/String;Ljava/lang/String;)Ljava/lang/String;
    .prologue
    const-string v0, ""
    return-object v0
```
**效果**：所有 HTTP POST 到 `121.14.211.49:8961`（死服务器）瞬间返回空，不再超时。

#### 6. initializeApp() stub（GameInterface.smali）

```smali
.method public static initializeApp(Landroid/app/Activity;)V
    .prologue
    return-void
```
**效果**：CMGame SDK（中国移动手机游戏）登录窗口不再弹出，20+ 死服务器 URL 不再访问。

#### 7. sendSms() stub + 购买直通（SmsSdk.smali + sanguoAndroid360.smali）

```smali
; sendSms() 开头插入:
return-void

; jifei() 中 sendSms() 后插入:
invoke-static {p0}, jifeiBack(I)V
const/4 v0, 0x1
invoke-static {v0}, nativePayResult(I)V
```
**效果**：点击购买按钮 → 不卡死 + 直接发放对应物品。

### Native/ARM（7 处）

#### 8. curl_easy_perform → 返回错误（libgame.so）

| 偏移 | 原始 | 修补 |
|:--|:--|:--|
| `0x558440` | `push {r4,r5,r6,lr}` | `movs r0, #7; bx lr` |

**效果**：所有 native libcurl HTTP 请求瞬间返回 `CURLE_COULDNT_CONNECT`。

#### 9. removeSpriteFramesFromFile → 跳过 plist 解析

| 偏移 | 原始 | 修补 |
|:--|:--|:--|
| `0x52C3C2` | `push {r4,r5,r6,r7,lr}` | `bx lr` |

**效果**：阻止 AniCartoon 清理时的 plist 解析，消除 SIGSEGV 堆损坏。

**注意**：此函数有两个入口——`0x52C3B0`（短 wrapper）和 `0x52C3C2`（真实函数体），必须 patch 后者。

**崩溃路径**（均被阻止）：
```
路径A: MapLayer::removeall → Hero::~Hero → AniCartoon::clear+244 → plist → 💥
路径B: MapLayer::removenpc → Npc::~Npc → AniCartoon::clear+150 → plist → 💥
```

#### 10. zhengbanlogic 状态 194 BGT → B（计数器分支）

| 偏移 | 原始 | 修补 |
|:--|:--|:--|
| `0x15EE9A` | `bgt #0x15eea0` | `b #0x15ee9c` |

**效果**：状态 194 中 counter > 10 时强制先执行安全函数再走场景切换，防止崩溃。

#### 11. 计数器始终为 0

| 偏移 | 原始 | 修补 |
|:--|:--|:--|
| `0x15EE94` | `adds r2, #1` | `movs r2, #0` |

**效果**：防止 counter > 10 触发异常路径。

#### 12. ZA 补丁——getNowMapIndex 正版门修复

| 偏移 | 原始 | 修补 | 效果 |
|:--|:--|:--|:--|
| `0x243EEE` | `B #0x243F28` (原 Z1) | `BEQ #0x243F28` | 恢复 zhengbanFlag 检查 |
| `0x243EFE` | `movs r0, #0x28` | `B #0x243F28` | step≠0 走正常索引 |

```c
// 等效 C 代码:
if (zhengbanFlag != 0) {
    if (zhengbanStep == 0) return 39;   // 序章（允许）
    else goto normal_path;              // 其他正版关卡→正常
}
// normal_path: 根据 pNowLevelID 计算
```

#### 13-14. Z2+Z3（已回退）

`zhengbanFlag` 存档默认值：两次 `MOVS R2, #1`。

## 未解决问题详解

### 1. "木牛流马"处黑屏

**现象**：游戏到达某个特定游戏机制（木牛流马）后，场景变为全黑。游戏循环仍在运行（"暂停"覆盖层正常），音频正常播放。

**分析**：此问题与 plist 解析崩溃无关（P1 已阻止崩溃）。黑屏说明场景中存在空渲染——init/restart 创建了场景但内容未正确加载。可能原因：
- 该机制需要服务端数据（已死）
- 相关资源文件缺失/损坏
- 特定关卡 ID 在 `MapLayer::restart()` 中走 > 38 的分支，而该分支未正确初始化所有必要数据

### 2. 二次进入关卡人物卡死

**现象**：返回大地图后再次进入第一关，人物无法移动，但 UI 正常显示。

**分析**：游戏状态未正确重置。可能原因：
- `zhengbanlogic` 状态机状态残留（state 194 的 counter 在退出时未重置）
- 关卡重新加载时某些全局标志未清除
- `MapLayer::restart()` 之前的状态清理不完全

### 3. 序章缺失

**现象**：游戏直接从第一关开始，跳过了原始的新手教程/序章关卡 (Level 39, `assets/MAP39.bin`)。

**分析**：经 IDA 反汇编分析，序章由 `HeroShuXing::getNowMapIndex()` 中的 zhengban 机制控制：
- `zhengbanFlag==1 && zhengbanStep==0` → 返回 Level 39（序章）
- Z2+Z3 将 zhengbanFlag 默认值从 `true` 改为 `false`，导致序章被跳过
- 回退 Z2+Z3 后序章仍未出现——原因可能是 `zhengbanFlag` 在 loadZB() 中被 UI_DEBUG 强制置零，或 zhengbanStep 在 getNowMapIndex 调用前已被设为非零值

## 关键技术要点

### 为什么 targetSdkVersion=22 是正确的选择

`requestLegacyExternalStorage` 在 Android 11 上对 **targetSdkVersion ≤ 29** 的所有应用生效，不是只有 =29。22 (Android 5.1) 位于运行时权限 (API 23) 之前，可同时获得：
- 旧版存储模型（requestLegacyExternalStorage 生效）
- 权限自动授予（无需手动操作）
- getDeviceId()/getSubscriberId() 无 SecurityException
- Apache HTTP 自动可用
- Non-SDK 接口不受限

### libcurl 在 Native 层的双重 HTTP

Java 层和 Native 层各自做 HTTP——前者通过 DefaultHttpClient，后者通过内嵌的 libcurl 7.26.0。两者都必须 stub。

### zhengbanlogic 是 30+ 状态的状态机

不是简单的计数器函数。状态 194 (0xC2) 只是其中的场景切换状态。计数器 block 只是状态 194 的一个分支。
