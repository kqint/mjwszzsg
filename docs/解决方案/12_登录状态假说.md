# 登录状态假说

## P1 失败分析

P1 成功阻止了 plist 解析崩溃，但黑屏依然存在。**plist 崩溃只是原版闪退的症状，不是黑屏的根因。**

## 新发现：sanguoAndroid360 的 JNI 回调字段

native 代码通过 JNI 调用 `sanguoAndroid360` 的静态方法，读写以下字段：

| 字段 | 默认值 | 写入方法 | 含义 |
|:--|:--|:--|:--|
| `denglu` | -1 | `dengluBack(int)` | **登录状态** |
| `guanka` | -1 | `guankaBack(int)` | **关卡/进度** |
| `jifei` | 0 | `jifeiBack(int)` | 计费结果 |
| `zbflag` | -1 | — | 正版标记 |
| `jinqian` | -1 | — | 金币 |

native 代码通过 JNI 查找这些 Java 方法（symbol 中找到 `checkSIM`, `jifei`, `jifeiBack`, `guankaBack`, `dengluBack`）：

```
native→Java 调用:
  checkSIM() → 返回 -1（始终，无 SIM 卡）
  jifei(int id) → 发短信计费 → SmsSdk.sendSms()
  jifeiBack(int id) → 计费结果回调 → 存储到 jifei 字段
  guankaBack(int id) → 关卡回调 → 存储到 guanka 字段
  dengluBack(int id) → 登录回调 → 存储到 denglu 字段
  Backflag() → 上传用户数据 → SmsSdk.sendUserData()
```

## 假说

**`denglu`（登录）字段是黑屏的关键。**

原始流程：
```
CMGame SDK → 用户登录 → dengluBack(0) → denglu = 0 → native 读到"已登录"
```

修补后流程：
```
initializeApp() → return-void → dengluBack() 永不调用 → denglu = -1
→ native 读到"未登录" → 到达登录门（固定进度）→ 拒绝继续 → 黑屏
```

前 10 次场景切换（试玩阶段）不检查登录状态。到达"登录门"后，游戏检查 denglu 字段：
- denglu ≥ 0 → 放行
- denglu = -1 → 阻塞 → 黑屏（或原版中显示登录窗口）

## 验证方案

### 方案 L1：直接设置 denglu 字段

在 `sanguoAndroid360` 的静态初始化 (`<clinit>`) 中，将 `denglu` 设为 0（表示已登录）：

```smali
; 在 <clinit>()V 中
sput v1, Lcom/gamedo/v360/sanguoAndroid360;->denglu:I  ; v1 = -1, 原值
; 改为:
const/4 v0, 0x0
sput v0, Lcom/gamedo/v360/sanguoAndroid360;->denglu:I  ; denglu = 0 (已登录)
```

### 方案 L2：Patch dengluBack 在 native 中的调用点

找到 native 代码中调用 `dengluBack` 的位置，将其改为 NOP，防止它把 denglu 改回 -1。

### 方案 L3：在 onCreate 中强制设置 denglu=0

在 `sanguoAndroid360.onCreate()` 中，执行完 InitSmsSdk 后，添加：
```smali
const/4 v0, 0x0
sput v0, Lcom/gamedo/v360/sanguoAndroid360;->denglu:I
```

同时设置 `guanka=1`（允许进入第一关）。

## 结果

❌ **L1 无效**。denglu=0 和 guanka=0 后，第一次启动仍闪退（Npc 路径的 plist 崩溃），后续启动不闪退但进度黑屏依旧。

登录状态假说可能不成立——或者需要修改更多字段（如 jifei、zbflag、jinqian 等也都是 -1）。

## 建议

L1 尝试证明了 denglu 字段不是单一关键开关。可能需要分析 native 代码具体检查了哪些字段，或者接受 H1 方案（有画面无 UI）寻找替代 UI 路径。
