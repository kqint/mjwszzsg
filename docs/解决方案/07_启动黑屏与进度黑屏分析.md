## 问题1：启动黑屏（141 秒）

### 时间线（从日志4提取）

```
23:27:03.850  进程创建 (am_proc_start)
23:27:03.896  BindApplication (Application.onCreate) - 极快，46ms
23:27:03.937  OplusCustomizeRestrictionManager init - 正常
              ↓
              ← 141 秒空白期，游戏进程无任何日志输出 ←
              ↓
23:29:24.833  SlowBinder: ITelephony code=153, cost=10215ms  ← 仅 10s
23:29:24.838  SELinux: denied iphonesubinfo1
23:29:24.864  onCreate 开始
23:29:24.904  阻塞报告: LaunchActivityItem 处理耗时 141004ms
```

### 关键发现

**1. 延迟发生在 performLaunchActivity 内部**

```
ActivityThread.main()
  → Looper.loop()
    → handleMessage(LAUNCH_ACTIVITY)
      → performLaunchActivity()     ← 整个 141 秒消耗在这里
        → activity.onCreate()       ← 包括 super.onCreate + SmsSdk.InitSmsSdk
```

**2. SlowBinder 只占 10 秒，剩余 131 秒无法解释**

唯一的游戏进程 SlowBinder：
```
com.android.internal.telephony.ITelephony code=153 cost=10215ms
```

ITelephony 是 `TelephonyManager` 的隐藏接口。code=153 是一个高编号方法（在 Android 11 上可能对应 `getImeiForSlot` 或类似方法）。这个调用不在我们修补的 4 个 SmsSdk 方法之内——它是其他 SDK 或 Cocos2dx 引擎代码触发的。

**3. SELinux 阻断 iphonesubinfo1 服务**

```
avc: denied { find } for name=iphonesubinfo1
scontext=u:r:untrusted_app_25:s0:c512,c768
```

`untrusted_app_25` 域的应用无法查找 `iphonesubinfo1`（IPhoneSubInfo）服务。这个服务是 `getDeviceId()` / `getSubscriberId()` 的底层实现。虽然我们在 SmsSdk 中 patch 了 4 个直接调用，但其他 SDK 组件中还有 20+ 个未修补的调用点——这些调用尝试查找服务时被 SELinux 阻断，binder 调用挂起直到超时。

### 根因

**141 秒 = 多个 binder 调用因 SELinux 阻断而超时**

即使是已修补的 SmsSdk 调用（我们替换了 `getDeviceId()` 等 4 个），启动链路中还有其他 SDK 初始化代码调用 telephony API：

| SDK 组件 | 调用 | 估计超时 |
|:--|:--|:--|
| 360 safiap SDK | `getDeviceId()`, `getSubscriberId()` | 30s |
| CMGame billing SDK | `getDeviceId()`, `getSubscriberId()` | 30s |
| 咪咕 mmwlan SDK | `getDeviceId()` (5+ 处) | 30s |
| 渠道 SDK (com.a) | `getDeviceId()`, `getSubscriberId()` | 30s |
| ITelephony code=153 | 未知 TelephonyManager 内部方法 | 10s |
| HTTP 请求 | SmsSdk.sendGameInfo() 连接死服务器 | 10-30s |

即使每次超时 10-30 秒，累积起来就是 2+ 分钟。

### 为什么 SELinux 阻断而不直接抛 SecurityException？

Android 框架有两个安全层：
1. **Java 层权限检查**：targetSdkVersion=22 下，`READ_PHONE_STATE` 权限在安装时自动授予，Java 层不抛 SecurityException
2. **SELinux (内核层)** ：无论 Java 层权限如何，SELinux 策略阻止 `untrusted_app_25` 域访问 `iphonesubinfo1` 服务

Java 层的调用通过了权限检查，进入 binder IPC 层，但 SELinux 在内核层拦截了服务查找。binder 调用阻塞直到超时（~30 秒），而不是快速失败。

### 为什么之前（日志3）的 SmsSdk patch 没帮助？

日志3 显示我们没有 patch SmsSdk（那还是原始状态）。日志4 的 SmsSdk patch 后，4 个调用不再阻塞（从 `const-string` 绕过了），但其他 SDK 的调用仍在阻塞。131 秒的剩余延迟由其他 SDK 的 telephony 调用和可能的 HTTP 请求超时组成。

## 问题2：进度黑屏

### 当前 ARM patch 行为

```
偏移 0x0015EE9A: 原始 bgt #0x15eea0 → 现在 b #0x15eeb6

效果：无条件跳转到 0x15EEB6，跳过以下代码块：
  0x15EEA0: scene_state = 300          ← 设置场景状态
  0x15EEA8: counter = 0               ← 重置计数器
  0x15EEAA: removeall()               ← 清理旧场景
  0x15EEB2: init(0)                   ← 初始化新场景
  0x15EEB6: call 0x15f8c6            ← 工具函数
```

### 为什么导致黑屏

`zhengbanlogic()` 调用 `removeall()` + `init()` 是**正常的场景切换机制**。在前 10 次调用中（counter 1-10），这段代码每次都执行且工作正常。第 11 次调用时，原生 bug 在 `removeall()` 内部的 plist 解析中触发崩溃。

我们之前的 B patch 跳过 `removeall()` 也跳过了 `init()`——这意味着：
- **旧场景资源未释放**：地图对象、精灵帧残留
- **新场景未初始化**：init(0) 未调用，GL 渲染数据未准备
- **结果**：OpenGL 渲染循环继续运行但无有效场景数据 → 黑屏

### 关键洞察

`removeall()` + `init()` 在 counter 1-10 期间是**正常工作的**。崩溃只在 counter > 10 时发生。这意味着 counter 的值本身影响 `removeall()` 的行为——可能是通过累积的游戏状态。

**0x15f8c6 这个函数是关键**：
- counter 1-10：BGT 不跳转 → 调用 0x15f8c6 → 再执行 removeall() → **正常**
- counter > 10（原版）：BGT 跳转 → 跳过 0x15f8c6 → 直接 removeall() → **崩溃**

`0x15f8c6` 在 removeall 之前执行了某种准备/清理工作，使得 removeall 能安全执行。当 counter > 10 时，跳过这个准备工作导致 removeall 崩溃。

## 解决方案

### 启动黑屏：全量修补 telephony 调用

需要修补所有 SDK 组件中的 telephony 调用，不仅仅是 SmsSdk。有两种策略：

**策略 1：修补启动路径上的调用点（优先）**

根据 [03_方案B_全量Smali修补.md](03_方案B_全量Smali修补.md) 的清单，需要在以下文件中也做类似修补：

| 优先级 | 文件 | 调用 |
|:--|:--|:--|
| 高 | `cn/game189/sms/SMS.smali:918` | getDeviceId() |
| 高 | `com/a/f.smali:283,450` | getDeviceId(), getSubscriberId() |
| 高 | `com/ccit/mmwlan/b/a.smali:59,108` | getDeviceId() |
| 中 | 其他 SDK 组件中的调用点 | 各种 |

但全量修补工作量大，且难以确定哪些调用在启动路径上。

**策略 2：使用 Xposed/LSPosed hook（推荐作为备选）**

如果设备已 root 或安装了 LSPosed，可以用 hook 框架全局拦截 `TelephonyManager.getDeviceId()` 等调用，返回空字符串。一次配置，全局生效。

但这需要额外的设备配置。

**策略 3：分析 HTTP 请求超时（低难度高收益）**

`SmsSdk.sendGameInfo()` 中的 HTTP 请求可能也在超时。在 SmsSdk.smali 中找到 `sendGameInfo` 方法中的 HTTP 调用，将其替换为直接返回。这可以节省 10-30 秒。

### 进度黑屏：修正 ARM patch 策略

之前我们的 patch 从"跳过 removeall"到"保护 removeall 不崩溃"的转换。正确策略：

**方案 G1：恢复 BGT 条件，但始终执行安全函数**

将 BGT 改为 `B #0x15ee9c`（无条件跳转到安全函数调用），这样：
1. 始终先调用 0x15f8c6（安全准备工作）
2. 然后执行 scene_state=300, counter=0, removeall(), init(0)（正常场景切换）
3. 安全函数预处理可能防止 removeall 崩溃

这等价于强制 counter > 10 时也走 counter ≤ 10 的流程。

但风险是：如果 0x15f8c6 本身在 counter > 10 时无法安全执行（比如它读取的数据在 > 10 状态下不同），可能仍然崩溃。

**方案 G2：分析 0x15f8c6 函数**

反汇编 offset 0x15f8c6 处的函数，理解它做了什么准备操作。如果它是一个清理函数（比如释放特定缓存），我们可以单独在 removeall 之前插入它。

**方案 G3：在 removeall 内部修补崩溃函数**

直接修补 `CCDictionary::setObject` 中的崩溃点（offset 0x005019AC: `ldr r4, [r3, #0xc]`）。在崩溃指令之前添加 NULL 检查，如果 r3 无效则跳转到错误处理路径。

这需要更深入的 ARM 汇编修改（添加条件分支和错误处理代码），但因为 libgame.so 中没有空闲空间，可能需要段扩展。

**方案 G4：分析为什么 counter > 10 时 removeall 崩溃**

比较 counter ≤ 10 和 counter > 10 时进入 removeall 的差异。唯一区别是 0x15f8c6 是否被调用。如果确定了 0x15f8c6 提供了什么关键准备，就可以做到最小修补。

## 建议优先级

```
1. 启动黑屏: 修补 HTTP 请求超时 (sendGameInfo)          ← 工作量小-中
2. 进度黑屏: 尝试方案 G1 (B #0x15ee9c 而非 B #0x15eeb6) ← 工作量小
3. 启动黑屏: 全量修补其他 SDK 的 telephony 调用         ← 工作量大
4. 进度黑屏: 如果 G1 不行，分析 0x15f8c6 函数           ← 工作量中
```
