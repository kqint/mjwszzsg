# 核心问题分析

## 为什么三次修复尝试都失败了？

三次尝试的共同特点：**都将 targetSdkVersion 设为 29**。

```
targetSdkVersion = 29 一次性触发的破坏性变更：

┌─────────────────────────────────────────────────────┐
│ API 23 (Android 6.0)  → 运行时权限机制              │
│ API 24 (Android 7.0)  → 文件 URI 暴露限制           │
│ API 26 (Android 8.0)  → 后台执行限制                │
│ API 27 (Android 8.1)  → 多窗口强制可调整大小        │
│ API 28 (Android 9.0)  → Apache HTTP 移除            │
│                      → Non-SDK 接口灰名单           │
│ API 29 (Android 10)  → getDeviceId() 抛异常          │
│                      → 分区存储                      │
└─────────────────────────────────────────────────────┘
```

将 targetSdkVersion 从"无声明"直接提升到 29，相当于让一个为 API 8 时代编写的 APK 同时面对 6 个 Android 大版本的所有限制变更。涉及的 7 个 SDK 组件全部未做适配。

## 困局本质

| 需求 | 冲突点 |
|------|--------|
| 需要 `requestLegacyExternalStorage` 生效 | 必须 targetSdkVersion ≤ 29（满足） |
| 需要避免 getDeviceId() 崩溃 | 必须 targetSdkVersion ≤ 28（冲突！） |
| 需要避免运行时权限 | 最好 targetSdkVersion ≤ 22 |

**关键认识**：`requestLegacyExternalStorage` 在 Android 11 上对 **targetSdkVersion ≤ 29** 的所有应用都生效，不仅仅是 29！之前锁定 targetSdkVersion=29 是过度收窄了可选范围。
