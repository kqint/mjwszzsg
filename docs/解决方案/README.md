# 解决方案总览

## 推荐尝试顺序

```
方案C          方案A                    方案B           方案D
获取logcat  →  降低targetSdkVersion  →  全量Smali修补  →  装箱/虚拟机
(10分钟)      到22 (30分钟)            (数小时)          (兜底)
```

## 为什么是这个顺序？

### 先做方案C（获取 logcat）— 10 分钟

这是回报率最高的步骤。花 10 分钟获取崩溃日志，就能确定：
- 崩溃是 Java 层还是 Native 层？
- 具体崩溃在哪个文件的哪个方法？

有了这个信息，后续方案不再是"盲修"。**强烈建议在任何修补之前先做这一步。**

### 再试方案A（降低 targetSdkVersion）— 30 分钟

这是投入产出比最高的方案：
- 只需改 Manifest 一行（targetSdkVersion: 29 → 22）
- 撤销之前的 smali 修补（恢复原始状态）
- 重新打包测试

如果成功，**一次性解决全部 29+ 个 getDeviceId() 调用点、Apache HTTP 缺失、Non-SDK 接口限制、运行时权限等问题。** 之前的三次尝试都没试过这个方向。

### 不行再方案B（全量修补）— 数小时

如果方案A失败（原因是 Android 11 对 targetSdkVersion 22 的应用仍然强制某些限制），则回到 targetSdkVersion=29 路线，配合 logcat 精确定位，逐一修补实际触发崩溃的调用点。

### 最终兜底方案D — 0 代码修改

如果所有修改方案都失败（尤其是 Native 层兼容性问题），装箱方案直接运行原版 APK，零代码改动。

## 各方案核心对比

| 方案 | 修改量 | 预计耗时 | 成功率 | 副作用 |
|:--|:--|:--|:--|:--|
| A: 降 targetSdkVersion | 1 行 | 30 分钟 | 较高 | 无 |
| B: 全量 Smali | 24+ 处 | 数小时 | 较高（Java层） | 计费/统计功能降级 |
| C: 抓 logcat | 0 | 10 分钟 | —（诊断手段） | 无 |
| D: 装箱 | 0 | 10 分钟 | 确定（原始APK） | 性能损耗/需网络 |

## 关键洞察

> **前三次尝试都失败的原因，不是修补不够多，而是 targetSdkVersion=29 这个选择本身。**
>
> 它一次性引入了 API 23 到 API 29 的所有行为变更。对 2014 年的代码来说，这相当于要求一个 Windows XP 程序不经修改直接跑在 Windows 11 上。
>
> targetSdkVersion=22 可能在 `requestLegacyExternalStorage` 生效的前提下，避开大部分陷阱。
> 这是整个修复历程中**唯一未尝试的有希望方向**。
