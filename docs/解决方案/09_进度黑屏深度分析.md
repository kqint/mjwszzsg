# 进度黑屏深度分析

## 当前状态

- 启动黑屏已完全修复（416ms 启动）
- 进度黑屏：H1/H3 均未解决

## 所有已尝试的 ARM patch

### 汇总表

| 方案 | 偏移 | 修改 | 效果 |
|:--|:--|:--|:--|
| NOP-BGT | `0x15EE9A`: `bgt`→`nop` | 安全函数始终执行 → 再落入 crash path | ❌ 进度黑屏 |
| B-skip | `0x15EE9A`: `bgt`→`b #0x15eeb6` | 跳过 removeall/init，直接到安全函数 | ❌ 进度黑屏 |
| B-safe | `0x15EE9A`: `bgt`→`b #0x15ee9c` | 安全函数→crash path(含removeall+init) | ❌ 进度黑屏 |
| **H1** | `0x15FAC0`: `bl zhengbanlogic`→`nop;nop` | 完全跳过 zhengbanlogic | ⚠️ 有画面，无UI，无交互 |
| **H3** | `0x15EE94`: `adds r2,#1`→`movs r2,#0` + revert H1 | 计数器始终为0，zhengbanlogic 完整运行 | ❌ 进度黑屏 |

### 关键观察

1. **H1 是唯一显示不同结果的方案**：游戏画面可渲染（人物、背景、动画、声音），但没有 UI、无法交互、进度不推进
2. **所有保留 removeall+init 的方案**都导致进度黑屏
3. **H3（计数器始终为0）**也黑屏——说明问题不在于 counter > 10 这个阈值

### 从 H1 结果得出的结论

zhengbanlogic() 做了两件事：
- **A. 场景初始化**：removeall() + init(0) —— 销毁旧场景，创建新场景
- **B. UI/交互初始化**：其他代码路径（函数A/函数B等）

H1 完全跳过后：B 缺失（无UI无交互），A 也缺失（当前场景保留，有人物背景）

### init(0) 的问题

`init(0)` 在所有 counter 值（1-10）时都正常工作，但在特定游戏进度点（第 11 次场景切换）创建黑屏。说明：
- `init(0)` 的行为可能依赖全局游戏状态
- 或者第 11 次场景切换需要不同的参数

## 用户假说：服务器数据缺失

### 假说内容

游戏依赖服务器提供用户存档数据（等级、道具、进度等）。我们删除了所有网络连接后：
- 用户没有"已登录"状态
- 没有云端存档
- 本地也没有写入存档的代码路径（因为原本依赖服务端）

`init(0)` 在创建新场景时检查登录状态/存档数据：
- Counter 1-10（试玩阶段）：允许无登录运行，使用默认数据
- Counter 11（正式游戏）：要求登录，检查存档 → 缺失 → 创建空场景

### 支持证据

1. SmsSdk.init() 原本发送设备信息到服务器，获取 `billtype` 和用户状态
2. CMGame SDK 原本提供登录认证（`gmc.g188.net/egsb/access/login`）
3. sendGameInfo 返回的 JSON 包含 `result.billtype`，决定计费模式
4. 存档可能通过 `UserInterface.aspx` 或 `UserDoneBillInterface.aspx` 上传/下载
5. 原版游戏在 VMOS 4.4 上也黑屏——服务器已死，所有用户都无法登录

### 可能涉及的数据

| URL | 数据方向 | 可能的数据 |
|:--|:--|:--|
| `SingleInterfaceV2.aspx` | 上行 | 设备信息（后台线程） |
| `GetBillTypeByICCID.aspx` | 下行 | 计费类型（`result.billtype`） |
| `UserInterface.aspx` | 上行+下行? | 用户数据/存档 |
| `UserDoneBillInterface.aspx` | 上行 | 计费完成通知 |
| `gmc.g188.net/egsb/access/login` | 下行 | 用户 token、登录状态 |

## 待验证的新方向

### 方向 1：分析 init(0) 函数的行为

反汇编 `0x15da90` 处的 init() 函数，理解参数 r1=0 的含义：
- 它是否检查全局"已登录"标志？
- 它是否读取 SmsSdk 的静态字段（如 operatorId）？
- 它是纯 native 函数还是通过 JNI 回调 Java？

### 方向 2：追踪 JNI 回调

检查 Cocos2dxHelper/Cocos2dxActivity 中是否有 JNI 方法被 init() 回调，这些方法可能读取 Java 层的用户状态。

### 方向 3：分析 SmsSdk.init() 中对 native 的影响

`SmsSdk.init()` 设置了多个静态字段（operatorId, imsi, imei, telephoneNumber 等）。检查这些字段是否通过 JNI 被 native 代码读取：
- `sanguoAndroid360` 中的 native 方法：`nativePayResult(int)`, `nativeUserBillSwitch(int)`
- `libidentifyapp.so` 中的 `getAppSignature()`

### 方向 4：尝试 B-skip 方案再次验证

`B #0x15eeb6` 方案跳过 removeall/init 但保留 zhengbanlogic 的其他逻辑。之前在无网络修复的情况下测试失败。现在网络已修复，可以重新测试。

### 方向 5：Mock 服务端响应

让 `sendGameInfo()` 返回一个模拟的有效 JSON（而非空字符串），例如：
```json
{"result":{"billtype":"1"},"status":"ok"}
```
观察是否能改变游戏行为。
