### 已确认的事实

1. **`removeSpriteFramesFromFile` 有两个入口**：
   - `0x0052C3B0`：短 wrapper (push {r4,lr} → pop {r4,pc}) — 我们第一次错误 patch 的
   - `0x0052C3C2`：真正的函数 (push {r4,r5,r6,r7,lr}) — 第二次正确 patch 的

2. **plist 解析崩溃有至少两条路径**：
   - 路径A：`MapLayer::zhengbanlogic() → removeall() → Hero::~Hero() → AniCartoon::clear()+244`
   - 路径B：`MapLayer::logic() → removenpc() → Npc::~Npc() → AniCartoon::clear()+150`

3. **P1（正确位置）阻止了 plist 解析，但黑屏依然存在**

4. **H1 是唯一有效的方案**——跳过 zhengbanlogic 后画面正常渲染（人物、背景、动画、声音），但无 UI 无交互

5. **黑屏 ≠ 崩溃**：修补版从不闪退，游戏循环/音频持续运行

## 核心矛盾

```
zhengbanlogic 运行 → UI/交互正常 → 但触发场景切换 → init(0)创建空场景 → 黑屏
zhengbanlogic 不运行 → 场景正常渲染 → 但无UI无交互 → 无法游玩
```

zhengbanlogic 做了两件互相冲突的事：
- **好事**：初始化 UI 和输入处理（必需）
- **坏事**：触发 removeall + init(0)（导致黑屏）

我们无法在不丢失 UI 的前提下阻止场景切换，也无法在保留场景切换的前提下修复空场景问题。

## 为什么所有 Native patch 都无效

`removeall()` + `init(0)` 是 zhengbanlogic 的**核心逻辑**——它们不是 bug，是正常的场景切换机制。前 10 次都能正常切换。

问题在于：在特定游戏进度（登录门），`init(0)` 需要的资源/数据不存在（因为：
- 服务器已死，无法提供用户数据
- 或游戏资产损坏
- 或需要特定的"已登录"标记，而我们的 denglu=0 可能不够

无论哪种原因，`init(0)` 创建了空场景 → 黑屏。我们在 native 层的所有 patch 都无法改变 `init(0)` 的行为。

## 剩余可行方向

### 方向 A：H1 + 部分 zhengbanlogic 恢复

保持 H1（跳过 zhengbanlogic），但找到 zhengbanlogic 中负责 UI 初始化的**具体状态**，只执行那些状态。

难度：高。需要逐个分析 30+ 状态处理器的功能。

### 方向 B：让 init(0) 加载有效场景数据

找到 init(0) 内部读取的全局状态/文件，提供有效的默认值。

难度：高。需要深入反汇编 init() 的数据加载逻辑。

### 方向 C：接受 H1 方案，寻找其他 UI 路径

如果 UI 层（ButtonLayer、ZDUILayer、PauseLayer 等）的初始化不依赖 zhengbanlogic 的状态机，可以单独调用它们。

难度：中。需要找到 UI 层的独立初始化方法。

### 方向 D：装箱方案

使用 VMOS Pro 或其他虚拟机运行原版 APK。虚拟机内是 Android 4.4-7.1，完全兼容该游戏。

难度：低。不需要修改 APK。

### 方向 E：寻找并修复 init(0) 内部依赖数据

init(0) 内部调用链可能涉及：
- 读取本地存档文件（SaveManager）
- 检查 JNI 回调的 Java 字段（denglu、guanka、jifei 等）
- 加载 plist/纹理资源

通过反汇编 init() 完整逻辑，确定它依赖的具体数据，然后确保这些数据可用。

难度：高。init() 函数约 300+ 指令。

## 建议

在这个阶段，最务实的方案是：

1. **短期**：装箱方案（VMOS）——游戏可以完整运行
2. **中期**：方向 A 或 E —— 需要更多反汇编分析
3. **放弃当前路线**：继续在 zhengbanlogic 的 counter/状态机层面 patch 已无意义——黑屏根因在 init(0) 的数据依赖
