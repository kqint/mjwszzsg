# 原版对比分析与最终方向

## 关键发现：原版 APK 对比修补版

### 原版 (日志7)

```
03:00:03  game CPU: user=28810 kernel=7980    ← 正常运行
   ↓ ~47秒，CPU 增加，游戏在加载内容
03:00:50  game CPU: user=36970 kernel=6360    ← 加载更多，进度推进了！
   ↓ ~2分23秒
03:03:13  Fatal signal 11 (SIGSEGV)           ← 最终还是崩溃
          fault addr 0x392c353d                ← 不同于之前的 0x7b2c7e3b
```

**原版偶尔能加载更多内容**，但最终仍以 SIGSEGV 崩溃。崩溃地址不同（`0x392c353d` vs `0x7b2c7e3b`），说明是**随机堆损坏**——不是固定的逻辑错误，而是内存破坏在不同运行时表现不同。

### 修补版 (日志8)

```
03:35:35  cocos2d-x debug info: ---80          ← 游戏逻辑在运行
          cocos2d-x debug info: ---15          ← 状态机值
          cocos2d-x debug info: ---10
          cocos2d-x debug info: ---9,10
          cocos2d-x debug info: lenght=27      ← 数组长度
          无崩溃，无 FATAL，无 tombstone
          Dolby 检测游戏在播放音频               ← 声音在播放
```

修补版不崩溃，游戏逻辑和音频正常，但画面黑屏。

## 综合分析

### 根因是同一个：CCDictionary::setObject() 中的堆损坏

```
原版:
  堆损坏 → SIGSEGV → App 闪退
  偶尔能多加载一些内容（取决于堆布局）

修补版:
  堆损坏 → OpenGL 状态被污染（非致命）
  → 纹理/缓冲区数据被破坏 → 渲染黑屏
  → 游戏逻辑和音频不受影响（在堆的不同区域）

为什么修补版不崩溃？
  - 我们的 patch 改变了内存分配模式
    (curl_easy_perform 不分配内存, counter=0 减少分配)
  - 堆布局不同 → 损坏发生在 GL 资源区而非代码指针区
  - 结果：GL 渲染损坏（黑屏）而非 SIGSEGV（闪退）
```

### 这个 bug 的特征

| 特征 | 说明 |
|:--|:--|
| 随机性 | 崩溃地址每次不同（0x7b2c7e3b, 0x392c353d） |
| 触发时机 | 固定游戏进度（特定场景切换） |
| 调用路径 | MapLayer::removeall() → CCArray::removeAllObjects → Hero::~Hero → AniCartoon::clear → CCSpriteFrameCache::removeSpriteFramesFromFile → plist 解析 |
| 崩溃位置 | CCDictionary::setObject(std::string const&) 的 string 操作 |

### 为什么我们之前的 patch 全部无效

| Patch | 预期 | 实际 | 原因 |
|:--|:--|:--|:--|
| NOP BGT | 跳过条件分支 | 黑屏 | 仍执行 removeall → 仍触发 bug → GL 损坏 |
| B #0x15eeb6 | 跳过 removeall/init | 黑屏 | 跳过了但 GL 状态已被之前操作污染 |
| B #0x15ee9c | 安全函数 → crash path | 黑屏 | removeall 仍执行 → GL 损坏 |
| H1 (skip zhengbanlogic) | 跳过整段 | 有画面无UI | 不执行 removeall → 场景保留，但无 UI 初始化 |
| H3 (counter=0) | 避免 >10 路径 | 黑屏 | removeall 仍执行（每次都执行） |
| N1 (curl_easy stub) | 消除 native HTTP | 黑屏 | 改变堆布局但 removeall 仍触发 bug |

**核心结论**：只要 `removeall()` 执行，bug 就会被触发。唯一不触发 bug 的方案是 H1（完全跳过 zhengbanlogic），但 H1 导致无 UI。

## 新方向：阻止 removeall() 内部的 plist 解析

### 崩溃链的精确位置

```
MapLayer::removeall()+782           (0x00157037)
  → CCArray::removeAllObjects()
    → Hero::~Hero()
      → AniCartoon::~AniCartoon()+44 (0x0016F0F9)
        → AniCartoon::clear()+244   (0x0016F001)
          → CCSpriteFrameCache::removeSpriteFramesFromFile()+26 (0x0052C3DD)
            → CCDictionary::createWithContentsOfFileThreadSafe()+28 (0x00501DBB)
              → CCDictionary::createWithContentsOfFile("plist文件")  [0x00522B03]
                → CCSAXParser::parse(plist) → 💥
```

### 方案 P1：patch CCSpriteFrameCache::removeSpriteFramesFromFile 直接返回

**文件**：`lib/armeabi/libgame.so`  
**偏移**：`0x0052C3B0`（函数入口）

```asm
; 函数签名: CCSpriteFrameCache::removeSpriteFramesFromFile(char const*)
; 偏移 0x0052C3B0:
  push {r4, lr}         ; 原始入口 (10 b5)
  adds r4, r0, #0       ;             (04 1c)
  bl   #0x52c368        ;             (ff f7 d8 ff)
  ...
  pop  {r4, pc}         ; 返回

; 修补后:
  bx   lr               ; 直接返回 (70 47)
```

效果：`AniCartoon::clear()` 调用 `removeSpriteFramesFromFile()` 时直接返回，不执行 plist 解析 → 不触发堆损坏 → 不崩溃也不黑屏。

**副作用**：精灵帧缓存不被清理，可能导致内存泄漏（但单次游戏影响小）。

### 方案 P2：patch AniCartoon::clear() 跳过 removeSpriteFramesFromFile 调用

在 `AniCartoon::clear()+244` (0x0016F001) 附近找到对 `removeSpriteFramesFromFile` 的调用，NOP 掉。

比 P1 更精准——只影响 AniCartoon 清理，不影响其他对 `removeSpriteFramesFromFile` 的合法调用。

### 方案 P3：patch CCDictionary::setObject() 的崩溃指令

在崩溃点 (0x005019AC: `ldr r4, [r3, #0xc]`) 之前添加 NULL 检查。但如果 r3 指向的是已释放但非 NULL 的内存（野指针），NULL 检查无法捕获。

### 方案 P4：恢复 H1 + 分离 UI 初始化

H1（跳过 zhengbanlogic）已经证明可以不触发 removeall 且显示场景。问题是缺少 UI 初始化。

如果我们能找到 zhengbanlogic 中负责 UI 初始化的具体状态处理器（ButtonLayer::iinit, ZDUILayer::iinit 等对应的状态），并只执行那些状态，就能既保留 UI 又不触发 removeall。

这需要更精细的状态机分析。

## 建议实施顺序

```
1. 方案 P1 (patch removeSpriteFramesFromFile → bx lr)     ← 最快，2字节
2. 如果 P1 不够 → 方案 P2 (精确 NOP AniCartoon 中的调用)
3. 备选 → 方案 P4 (恢复 H1 + 逐步恢复 UI 状态)
```

## 总结：五个阶段的完整修复历程

| 阶段 | 问题 | 根因 | 修复 | 结果 |
|:--|:--|:--|:--|:--|
| 1 | 闪退 | targetSdkVersion=29 触发 SecurityException | → targetSdkVersion=22 | Java 层全通 |
| 2 | 闪退 | NetworkOnMainThreadException | StrictMode.permitAll() | 修复 |
| 3 | 启动黑屏 141s | HTTP 到死服务器超时 | sendGameInfo→return "" / initializeApp→return | **启动 0.4s** |
| 4 | 进度闪退 | removeall() 堆损坏 → SIGSEGV | ARM B #0x15ee9c + H3 counter=0 | 不闪退 → 转为黑屏 |
| 5 | 进度黑屏 | 同根因，GL 状态被污染而非崩溃 | **P1/P2 (待实施)** | 待验证 |
